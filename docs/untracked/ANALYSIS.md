# Анализ требований: Сервис доставки готового питания для физических лиц

## 1. Резюме разговора (транскрибация)

### Участники
- **Заказчик** (женщина) — руководитель/представитель компании, которая уже работает с юридическими лицами (кофейни), поставляя им готовую еду оптом через существующий сайт + AmoCRM + 1С.
- **Разработчики** (Мурат + коллега) — обсуждают MVP нового проекта.

### Контекст
Компания уже имеет B2B-бизнес: производство готовой еды → доставка в кофейни (юрлица). Сейчас хотят запустить **новый B2C-проект** — доставку готовой еды для физических лиц по подписке. Сайт m-food.ru используется как **референс** (образец), а не как связанный проект.

### Ключевые решения из разговора

#### 1.1. Продуктовая база
- Товары (блюда) **скорее всего будут идентичны** существующим для юрлиц, но **не точно** → решение: **отдельная учётная запись/база** в CRM, не смешивать с юрлицами.
- Более 80 позиций блюд в текущем ассортименте (отдельные блюда, не наборы).
- Фотографии блюд хранятся в CRM (AmoCRM), загружаются в карточку товара.
- Для нового проекта будет **отдельная CRM-база** (новый аккаунт AmoCRM), чтобы данные B2B и B2C не смешивались.

#### 1.2. Концепция продукта
- **Только линейка "Просто"** — обычная домашняя еда. Без фит-линеек, спорт-линеек, лайт и т.д.
- Позиционирование: "просто еда, чтобы не готовить" — не для спортсменов/худеющих.
- Конкурентное отличие: конструктор — клиент сам выбирает блюда, а не получает фиксированный рацион.

#### 1.3. Модель заказа (подписка)
- Клиент **оплачивает подписку вперёд** на выбранный период (пакет/тариф).
- Тарифы: начально **3 штуки** (по аналогии с m-food: 3 дня, 6 дней, 30 дней), но в будущем могут добавляться.
- Нужна возможность **добавлять, редактировать, удалять тарифы** через админку.
- Компания предоставляет **образцовое меню на неделю**, но клиент может заменять блюда из номенклатуры (конструктор).
- Доставка: **2 раза в неделю** (еда на 3 дня за раз). Пример: понедельник — доставка на пн/вт/ср, четверг — на чт/пт/сб.
- Дополнительные блюда: можно добавить к рациону за доплату.

#### 1.4. Авторизация и данные клиента
- Регистрация по **номеру телефона + email**.
- SMS-верификация.
- Обязательно: **договор оферты** и **согласие на обработку персональных данных** при регистрации.
- Юридическое лицо у компании есть.

#### 1.5. Оплата
- В существующем B2B-проекте оплата была по факту (отсрочка, акты сверки, реквизиты).
- Для B2C нужна **онлайн-оплата картой** (как на m-food.ru).
- Подписка оплачивается вперёд.

#### 1.6. Бонусы и лояльность
- **Бонусная система планируется**, но НЕ на первом этапе MVP.
- Возможно: реферальная ссылка, копление бонусов.
- Нужно **заложить архитектурно**, но реализовать позже.

#### 1.7. Дизайн
- Логотипа и фирменного стиля для нового проекта **пока нет**.
- Заказчик обещал подготовить: логотип, цветовую гамму (ориентир — этикетки на продукции).
- Ожидание: ~2.5 недели на подготовку бренда.

#### 1.8. Ожидания от MVP
- Сайт-витрина с представлением товаров/пакетов.
- Конструктор меню на неделю.
- Личный кабинет с заказами.
- Онлайн-оплата подписки.
- Админка для управления товарами, тарифами, клиентами (большая часть в CRM).

## 3. Требования к MVP бэкенда

### 3.1. Что входит в MVP (первый этап)

На основе разговора и анализа m-food.ru:

1. **Каталог блюд** — CRUD для блюд (админка)
2. **Программы/тарифы** — конфигурируемые тарифы (3/6/30 дней и т.д.)
3. **Конструктор меню** — формирование рациона на период с возможностью замены блюд
4. **Авторизация** — по номеру телефона (SMS) + email
5. **Профиль пользователя** — персональные данные, адреса доставки
6. **Оформление заказа (подписки)** — выбор тарифа → конструктор → чекаут → оплата
7. **Онлайн-оплата** — интеграция с платёжным шлюзом
8. **История заказов** — текущие и прошлые
9. **Промокоды** — применение скидок при оформлении

### 3.2. Что НЕ входит в MVP (последующие этапы)

- Бонусная система (кэшбэк, уровни)
- Реферальная программа ("Пригласить друга")
- Программа "Лайт" (еда по калориям)
- Страница акций
- Блог
- Мобильное приложение
- Интеграция с CRM (AmoCRM) — на MVP данные управляются через собственную админку
- Интеграция с 1С
- Пуш-уведомления / Email-рассылки

## 4. Архитектура бэкенда (Go)

### 4.1. Доменные сущности

```
User (Пользователь)
├── id: UUID
├── phone: string (уникальный, формат +7...)
├── email: string (nullable)
├── name: string (nullable)
├── birthday: date (nullable)
├── gender: enum (male/female/unknown)
├── password_hash: string (nullable, если только SMS-авторизация)
├── created_at: timestamp
└── updated_at: timestamp

Address (Адрес доставки)
├── id: UUID
├── user_id: UUID → User
├── street: string
├── building: string
├── entrance: string (nullable) — подъезд
├── floor: string (nullable) — этаж
├── apartment: string (nullable) — квартира
├── intercom: string (nullable) — домофон
├── is_default: bool
├── created_at: timestamp
└── updated_at: timestamp

Dish (Блюдо)
├── id: UUID
├── name: string
├── description: string (nullable)
├── image_url: string
├── weight_grams: int — вес порции
├── calories: int — Ккал
├── proteins: float — Белки (г)
├── fats: float — Жиры (г)
├── carbs: float — Углеводы (г)
├── category: enum (breakfast/lunch/dinner/snack)
├── is_active: bool
├── sort_order: int
├── created_at: timestamp
└── updated_at: timestamp

Plan (Тариф/Пакет)
├── id: UUID
├── name: string — "3 дня", "6 дней", "30 дней"
├── duration_days: int — 3, 6, 30
├── meals_per_day: int — количество блюд в день (3, 4, 5, 6)
├── price_per_day: decimal — цена за день (базовая, для 3 блюд)
├── discount_percent: int — скидка (0%, 8%, 47%)
├── is_active: bool
├── sort_order: int
├── created_at: timestamp
└── updated_at: timestamp

ExtraMealPrice (Цена доп. блюда)
├── id: UUID
├── plan_id: UUID → Plan
├── extra_meals_count: int — сколько доп. блюд
├── price_per_day: decimal — доплата за день
└── created_at: timestamp

Order (Заказ/Подписка)
├── id: UUID
├── user_id: UUID → User
├── plan_id: UUID → Plan
├── address_id: UUID → Address
├── status: enum (pending/paid/active/delivering/completed/cancelled)
├── meals_per_day: int — выбранное количество блюд
├── extra_meals: int — доп. блюда (0, 1, 2...)
├── start_date: date — дата начала
├── end_date: date — дата окончания
├── delivery_time_slot: string — "9:00-11:00"
├── total_price: decimal — итоговая цена
├── discount_amount: decimal — скидка (промокод)
├── promo_code_id: UUID → PromoCode (nullable)
├── comment: string (nullable) — комментарий к доставке
├── payment_id: string (nullable) — ID платежа в платёжном шлюзе
├── created_at: timestamp
└── updated_at: timestamp

OrderMenu (Меню заказа — конкретные блюда на каждый день)
├── id: UUID
├── order_id: UUID → Order
├── date: date — на какой день
├── meal_slot: enum (breakfast/lunch/dinner/snack1/snack2/snack3)
├── dish_id: UUID → Dish
├── is_replaced: bool — заменено ли пользователем
├── created_at: timestamp
└── updated_at: timestamp

Delivery (Доставка — физическая доставка, несколько за подписку)
├── id: UUID
├── order_id: UUID → Order
├── date: date — дата доставки
├── status: enum (planned/preparing/delivering/delivered/failed)
├── covers_dates: date[] — какие дни покрывает (напр. пн, вт, ср)
├── created_at: timestamp
└── updated_at: timestamp

PromoCode (Промокод)
├── id: UUID
├── code: string (уникальный)
├── discount_type: enum (percent/fixed)
├── discount_value: decimal — 15 (%) или 500 (₽)
├── max_uses: int (nullable) — лимит использований
├── used_count: int
├── valid_from: timestamp
├── valid_until: timestamp (nullable)
├── is_active: bool
├── created_at: timestamp
└── updated_at: timestamp

OTPCode (Код подтверждения SMS)
├── id: UUID
├── phone: string
├── code: string — 4-6 цифр
├── expires_at: timestamp
├── is_used: bool
├── attempts: int — количество попыток ввода
├── created_at: timestamp
└── updated_at: timestamp
```

### 4.2. API Endpoints (REST)

#### Авторизация
```
POST   /api/v1/auth/send-code        — Отправить SMS-код
POST   /api/v1/auth/verify-code       — Подтвердить код → получить JWT
POST   /api/v1/auth/refresh            — Обновить JWT-токен
POST   /api/v1/auth/logout             — Выход
```

#### Профиль пользователя
```
GET    /api/v1/profile                 — Получить профиль
PUT    /api/v1/profile                 — Обновить профиль (имя, email, день рождения, пол)
```

#### Адреса доставки
```
GET    /api/v1/addresses               — Список адресов
POST   /api/v1/addresses               — Добавить адрес
PUT    /api/v1/addresses/{id}          — Обновить адрес
DELETE /api/v1/addresses/{id}          — Удалить адрес
```

#### Каталог блюд
```
GET    /api/v1/dishes                  — Список активных блюд (с фильтрацией по категории)
GET    /api/v1/dishes/{id}             — Карточка блюда (КБЖУ, фото, вес)
```

#### Тарифы
```
GET    /api/v1/plans                   — Список активных тарифов с ценами
GET    /api/v1/plans/{id}              — Детали тарифа
```

#### Конструктор меню
```
GET    /api/v1/menu/default?plan_id=X&start_date=Y  — Образцовое меню на период
PUT    /api/v1/menu/customize          — Заменить блюдо в конкретный день/слот
```

#### Заказы
```
POST   /api/v1/orders                  — Создать заказ (тариф + меню + адрес + дата + слот)
GET    /api/v1/orders                  — Список заказов (текущие/прошлые)
GET    /api/v1/orders/{id}             — Детали заказа (включая меню по дням)
POST   /api/v1/orders/{id}/cancel      — Отменить заказ
```

#### Промокоды
```
POST   /api/v1/promo/validate          — Проверить промокод и получить сумму скидки
```

#### Оплата
```
POST   /api/v1/payments/create         — Инициировать платёж (→ URL платёжного шлюза)
POST   /api/v1/payments/callback       — Webhook от платёжного шлюза
GET    /api/v1/payments/{order_id}/status — Статус оплаты
```

#### Доставки (для просмотра клиентом)
```
GET    /api/v1/deliveries?order_id=X   — Список доставок по заказу
```

#### Админка (отдельный префикс, защищённый)
```
# Блюда
GET    /api/v1/admin/dishes            — Список всех блюд (включая неактивные)
POST   /api/v1/admin/dishes            — Создать блюдо
PUT    /api/v1/admin/dishes/{id}       — Обновить блюдо
DELETE /api/v1/admin/dishes/{id}       — Деактивировать блюдо
POST   /api/v1/admin/dishes/{id}/image — Загрузить фото блюда

# Тарифы
POST   /api/v1/admin/plans             — Создать тариф
PUT    /api/v1/admin/plans/{id}        — Обновить тариф
DELETE /api/v1/admin/plans/{id}        — Деактивировать тариф

# Промокоды
GET    /api/v1/admin/promo             — Список промокодов
POST   /api/v1/admin/promo             — Создать промокод
PUT    /api/v1/admin/promo/{id}        — Обновить промокод

# Заказы
GET    /api/v1/admin/orders            — Все заказы (фильтры: статус, дата, клиент)
PUT    /api/v1/admin/orders/{id}/status — Обновить статус заказа

# Доставки
GET    /api/v1/admin/deliveries        — Список доставок (фильтры: дата, статус)
PUT    /api/v1/admin/deliveries/{id}/status — Обновить статус доставки

# Дефолтное меню
PUT    /api/v1/admin/menu/default      — Задать образцовое меню на неделю

# Пользователи
GET    /api/v1/admin/users             — Список пользователей
GET    /api/v1/admin/users/{id}        — Детали пользователя
```

### 4.5. Ключевые бизнес-процессы

#### Процесс регистрации/авторизации
```
1. Клиент вводит номер телефона
2. Бэкенд генерирует OTP (4-6 цифр), сохраняет в БД с TTL 5 мин
3. Отправляет SMS через провайдер
4. Клиент вводит код
5. Бэкенд проверяет код → создаёт/находит пользователя → возвращает JWT
6. При первом входе показывается согласие на обработку ПД + оферта
```

#### Процесс оформления заказа
```
1. Клиент выбирает программу ("Просто")
2. Выбирает количество блюд (3/4/5/6)
3. Выбирает тариф (3/6/30 дней)
4. Видит дефолтное меню на период → может заменять блюда (конструктор)
5. Опционально: добавляет доп. блюда (+1 за N ₽/день)
6. Переходит на чекаут → заполняет/выбирает адрес, дату, слот времени
7. Применяет промокод (опционально)
8. Нажимает "Оформить заказ" → редирект на платёжный шлюз
9. После успешной оплаты → заказ переходит в статус "active"
10. Система автоматически генерирует расписание доставок (2 раза/нед)
```

#### Процесс доставки (логика)
```
Для подписки на N дней:
- Доставка 1: день 1 → еда на дни 1, 2, 3
- Доставка 2: день 4 → еда на дни 4, 5, 6
- ...и т.д.

Доставка планируется автоматически при создании заказа.
Статусы: planned → preparing → delivering → delivered / failed
```
